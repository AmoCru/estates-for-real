## Vercel Functions and Supabase

The Angular app currently calls a Vercel Function at `/api/projects/42/shell` through `src/app/core/api/project-api.service.ts`. The function reads from Supabase server-side and falls back to demo data when the Supabase environment variables are not configured.

Create these environment variables in Vercel under Project Settings > Environment Variables:

```bash
SUPABASE_URL=https://your-project-ref.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-server-only-service-role-key
SUPABASE_PROJECTS_TABLE=projects
```

For local function testing, put the same real values in `.env`. Keep `.env.example` as placeholders only and never commit a real Supabase service role key.

The default endpoint expects a Supabase table named `projects` with these columns: `id`, `name`, `phase`, `role`, and `scope`. The current demo project uses numeric id `42`, so the frontend and fallback data use the string route id `'42'` while the function queries Supabase with `id=eq.42`. If your table has another name, set `SUPABASE_PROJECTS_TABLE`. If your column names differ, update `api/projects/[projectId]/shell.js` to map them.

Current backend status:

1. `GET /api/projects/42/shell` returns the project shell, sidebar navigation, and three static shell metrics.
2. Supabase is used for the project shell fields only.
3. Navigation and metrics are still generated by the Vercel Function from local demo constants.
4. Angular falls back to `src/app/core/mock-backend/mock-project.service.ts` if the function is unavailable.
